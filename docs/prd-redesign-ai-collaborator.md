# 问津 Agent 产品重新设计 —— "AI 全程协作者" 提案

> 状态：**草案，待评审**。本文档不直接生效，评审通过后按文末「文档同步清单」合并进 `docs/backend-prd.md` / `docs/frontend-prd.md`。
> 目标读者：产品/研发自己（决策记录）+ 面试官（能看懂"这是个 Agent 产品"而不是"表单+报告生成器"）。

---

## 0. 问题陈述

现状体验路径：**填表单 → 等 loading → 看一张静态报告卡片 → 角落里有个答疑机器人**。

问题不是功能少，是**技术复杂度和产品体验脱节**：后端已经有 LangGraph 多节点并行编排、Reflection 自我修正循环、CircuitBreaker 熔断降级、ToolResponse 三态协议——这些都只在 `/admin/debug`（403 保护、普通用户和大多数面试官根本看不到）里可视化。用户侧和默认产品体验里，AI 是黑盒。

**新定位**：AI 不是后台跑一次就交卷的批处理任务，而是一个**过程可见、可打断、可对话调整**的协作者。用户在两处直接感知"AI 在做什么、AI 在怎么想"：

1. **生成过程本身**（方向一）：从"进度条"变成"能看懂的 Agent 协作直播"。
2. **拿到报告之后**（方向二）：从"一次性交付"变成"可以继续对话、修改约束、重新生成、对比方案"的持续协作。

---

## 1. 设计原则（新增，补充 frontend-prd.md §4）

| 原则 | 说明 |
| --- | --- |
| **过程可见** | 凡是后端已经产生的 Agent 协作信号（并行执行、自我修正、降级重试），只要不涉及技术黑话或用户 PII，就应该有一个用户能看懂的呈现方式，而不是只写进 Debug 事件白名单。 |
| **对话优先于表单** | 需要澄清歧义或收集补充信息时，优先用"AI 追问 + 用户回答"的对话式交互，而不是让用户对着一堆表单字段自己猜怎么填。表单只用于结构化、无歧义的字段（省份、分数这类）。 |
| **报告是起点不是终点** | 报告生成完成 ≠ 交互结束。用户应该能在报告基础上继续说"改一下约束""这个和那个比比看"，AI 能听懂意图并采取行动（重新生成/对比），而不是只能回答"为什么推荐这个"。 |
| **技术真实性** | 所有"看起来很智能"的呈现必须对应后端真实发生的事情，不能为了好看而编造不存在的 Agent 行为（这也是可以在面试里站得住脚的前提）。 |

### 1.1 关键决策：Generative UI，而非纯对话式

讨论过是否把整个产品做成纯聊天界面（类似千问高考把报告挂在通用 Chat 对话里），结论是**不做**：志愿填报是信息密集、需要反复横向比较的决策任务（几十条候选、冲稳保分层、位次差、风险标记），聊天记录是线性的、一次性的，无法满足"扫一眼定位、随时回看对比"的需求。这一点在竞品分析里也得到反证——千问高考把报告嵌进聊天对话，我们评估的结论是"专业操作性弱于我们的分层 Tab + 风险总览"，即纯聊天承载结构化决策信息是已验证的劣势，不是该学的地方。

**采用形态**：chat 是控制层，结构化视图（Tab、推荐卡片、风险总览、对比视图）是呈现层，两者不是两个割裂的目的地，而是同一个 Agent 交互的两种输出形式：

- 报告页不再是"生成完就定型的静态页面"，而是 Agent 可以持续操作的**画布**：用户说"预算改一下""帮我对比这两个"，ConversationAgent 除了回复文字，还能直接调用**前端 UI 操作工具**去更新画布上的具体组件（切换到对应 Tab、高亮某几张卡片、弹出对比视图），而不是只在聊天气泡里贴一段新文字。
- 用户始终有一个常驻的对话入口表达意图，但看数据、比方案时用的是结构化组件，不需要在聊天记录里翻找。

这个决策贯穿方向二的具体设计（§3.2、§3.3），把它们从"聊天触发 API → 返回新数据 → 用户手动切换查看"，进一步收紧为"聊天触发 → Agent 直接操作画布组件 → 用户所见即所得"。这也是可以在面试里讲清楚的产品判断：**主动评估过纯 Chat 方案并基于任务性质放弃，而不是无脑追随"全部聊天化"的潮流**。

## 2. 方向一：编排能力前端化（把"彩蛋"变成"主线体验"）

### 2.1 现状 vs 改造

| | 现状 | 改造后 |
| --- | --- | --- |
| 用户侧 SSE 事件 | `node_started` / `evidence_found` / `rule_checked` / `candidates_ready` / `risk_found` / `completed` / `error` | 新增 `agents_parallel_started` / `agents_parallel_merged` / `self_check_round` / `degraded_notice`（见 2.3） |
| 生成进度页（8.5） | 纵向 checklist，8 个步骤挨个转圈 | "Agent 协作时间线"组件：能看到哪几个 Agent 在**同时**工作、Reflection **自我修正了几轮**、遇到降级时**怎么兜底** |
| 报告详情页 | 报告生成完直接展示结果，过程信息丢弃 | 新增可折叠「AI 是如何得出这份方案的」卡片，复用生成过程的事件记录做**回放**，不是重新查询 |
| Admin Debug Console | 唯一能看到编排细节的地方 | 保留（技术纵深展示用，面试时可以直接演示这个当"我还做了更硬核的可观测性"的加分项），但**不再是用户唯一能感知 AI 复杂度的地方** |

### 2.2 生成进度页重新设计（替换 frontend-prd.md §8.5）

**布局调整**：把原来单一纵向 checklist 拆成两层信息：

```
┌─────────────────────────────────────────┐
│ 报告标题 + 考生简要信息                     │
├─────────────────────────────────────────┤
│ 【协作时间线】（主区域，取代原 checklist）    │
│                                           │
│  ● 档案检查 · 已完成                        │
│  ● 数据版本锁定 · 已完成                     │
│  ┌─ 正在并行处理 ──────────────────┐        │
│  │ ⟳ 检索招生数据与政策依据          │        │
│  │ ⟳ 校验选科/体检/批次规则          │        │
│  └────────────────────────────────┘        │
│    → 已发现 2026年河南省招生计划数据          │
│    → 选科校验通过，体检项发现 1 处限制         │
│  ● 生成候选方案 · 已完成（共 48 所候选）       │
│  ● 志愿梯度体检 · 已完成                      │
│  ● 生成报告初稿 · 已完成                      │
│  ┌─ AI 正在自我检查 ────────────────┐         │
│  │ 第 1 轮：发现表述可能构成过度承诺   │         │
│  │         → 正在修正...            │         │
│  │ 第 2 轮：检查通过 ✓               │         │
│  └────────────────────────────────┘         │
│  ● 报告已生成                                │
├─────────────────────────────────────────┤
│ 预计剩余时间 / 已用时                        │
└─────────────────────────────────────────┘
```

**关键改造点**：

1. **并行执行要显式画出来**，不能拍扁成两条顺序步骤。收到 `agents_parallel_started` 事件时，两个子任务用一个带缩进的分组框同时展示"进行中"状态；收到 `agents_parallel_merged` 后合并收起，保留结果摘要。
2. **Reflection 回退要让用户看见"AI 在自我修正"**，这是本次改造里性价比最高的一处——完全复用已有的 `reflection_iterations` / `compliance_issues` 数据，只是把它从"Debug 专属"提升为"用户可见但脱敏"。展示规则：只显示轮次和**类别化的修正原因**（"表述可能构成过度承诺"/"证据引用不完整"），不展示具体禁用词或原始 LLM 输出。
3. **降级要转译成安心文案，而不是技术错误**。`degraded` 事件原文类似 `vector_search→sql_search`，用户侧转译为"检索遇到延迟，已切换备用数据源，不影响结果质量"，正面框定而不是让用户担心出错了。
4. 失败处理沿用原规范（8.5 已有），不变。

### 2.3 新增用户侧 SSE 事件（补充 backend-prd.md §5.7）

| 事件 | 数据 | 转译来源 | 隐私约束 |
| --- | --- | --- | --- |
| `agents_parallel_started` | `{"agents": ["retrieval_agent", "policy_rule_agent"], "message": "正在同时检索数据和校验规则"}` | 复用 Debug 的 `parallel_fan_out`，去掉节点内部技术名，只保留友好文案 | 无 PII |
| `agents_parallel_merged` | `{"agents": [...], "summary": "证据检索完成，规则校验完成"}` | 复用 `parallel_fan_in` | 无 PII |
| `self_check_round` | `{"iteration": 1, "max_iterations": 3, "issue_category": "over_promise" \| "evidence_gap" \| "none", "status": "revising" \| "passed"}` | 复用 Reflection 的 `compliance_issues`，**只传类别枚举，不传原始文本** | 类别化，无原始违规文本 |
| `degraded_notice` | `{"stage": "retrieval", "message": "检索遇到延迟，已切换备用数据源"}` | 复用 `degraded` 事件，人工维护"技术降级原因 → 用户文案"映射表 | 不暴露具体服务名（如 Cohere/pgvector） |

> 这四个事件与现有白名单事件走同一条 `sse:{run_id}` Redis Stream，用户侧 SSE 生成器按新白名单转发即可，**不需要新的事件源**，改造成本主要在事件文案映射层和前端组件，后端只是把已经产生的信号多推一份"友好版"。

### 2.4 报告页新增「AI 是如何得出这份方案的」卡片

- 位置：报告详情页风险总览卡片下方，默认折叠。
- 内容：复用生成过程中记录的时间线数据（存入 `agent_runs.debug_summary_json` 或新增用户可见的 `run_summary_json` 字段），**只读回放**，不重新调用 Agent。展示节选：并行处理过哪些任务、Reflection 修正了几轮、是否发生过降级。
- 价值：把生成过程中转瞬即逝的"直播"信息沉淀成报告的一部分，用户事后想验证"这个报告靠不靠谱"时可以回来看，也是面试演示时的固定展示点（不用掐时间去等 loading 页面）。

## 3. 方向二：多轮交互能力（从"一次性交付"到"持续协作"）

### 3.1 建档问诊：从纯表单到"表单 + AI 追问"混合

**现状问题**：backend 已经设计了 Profile Agent 连续追问（`profile_pending_questions`，最多 3 轮），但前端 8.4 是完全静态的 6 步表单，这个能力完全没有被使用——用户输入矛盾或模糊信息（比如选了物理但再选科目组合在该省不存在、体检限制勾了"是"但没说具体是什么）时，现状是等到最后风险体检才告知，而不是当场澄清。

**改造**：

- 表单仍然保留（省份/分数/位次这类无歧义字段用表单最高效，不强行对话化）。
- 提交后端做轻量校验（复用 Policy Rule Agent 的规则工具做前置检查，不等于完整生成方案）：命中矛盾或模糊项时，返回 `profile_pending_questions`，前端在"下一步"位置插入一张**对话式追问卡片**（区别于表单字段，样式类似 Chat 气泡）：

```
🤖 你选择了"物理+化学+生物"，但河南省 2026 年该组合在你所在批次
   暂无对应招生计划。是否要调整选科组合，还是仍按此组合查看结果？
   [调整选科]  [仍按此组合继续]
```

- 追问轮数上限沿用后端已有约束（最多 3 轮，超出后带着 `data_warnings` 继续，前端展示"以下信息可能不完整"提示）。
- 验收标准（Given/When/Then）：
  - **Given** 用户选择的选科组合在目标省份/批次无历史招生数据，**When** 提交基本信息后，**Then** 系统在下一步展示 AI 追问卡片而不是直接进入下一个表单步骤。
  - **Given** 用户已达到 3 轮追问上限，**When** 仍存在未澄清项，**Then** 系统正常进入下一步，并在生成进度页/报告页明确标注"以下信息基于不完整档案生成"。

### 3.2 报告页对话面板升级：从"答疑"到"Agent 操作画布"

**现状**：ConversationAgent 只能回答关于报告的问题，无法响应"预算改到 8 万以内""这所学校别推荐了""帮我对比这两个方案"这类修改/操作意图，且目前是纯 streaming completion，不具备 tool-calling 能力。

**改造**：按 §1.1 的 Generative UI 决策，把 ConversationAgent 升级为具备两类工具的 tool-calling agent（LangGraph 工具调用范式，Retrieval Agent / Policy Rule Agent 已经在用，技术路径已验证，只是首次应用到 ConversationAgent 上）：

**工具分类**：

| 类型 | 工具示例 | 效果 | 是否触发重新计算 |
| --- | --- | --- | --- |
| UI 操作类 | `switch_tab(tier)` / `highlight_candidates(ids)` / `open_compare_view(a, b)` / `expand_risk_detail(id)` | 纯前端状态变更，画布上对应组件联动 | 否，前端本地状态即可完成 |
| 数据变更类 | `regenerate_recommendations(patch)` | 后端重新计算候选/风险/报告 | 是，见下方局部重跑规则 |

**交互流程**：

1. 用户输入意图后，ConversationAgent 判断该用哪类工具（也可以不调工具，纯文字回答）。判断结果不影响现有答疑体验，只在识别到明确操作/修改意图时调用工具。
2. **UI 操作类**直接执行，AI 回复里带一句确认文案（"已为你打开对比视图"），不需要用户二次确认——因为不涉及重新计算，可撤销、无成本。
3. **数据变更类**（`regenerate_recommendations`）识别到后，先在聊天气泡下方展示确认卡片再执行，避免误触发重新生成消耗成本：

   ```
   检测到你想调整：预算上限 → 8 万元/年
   [确认，重新生成方案]  [取消，继续问答]
   ```

   确认后调用新接口 `POST /api/v1/reports/{report_id}/refine`（新增，补充 backend-prd.md §5）：
   - **轻量约束**（预算、城市偏好、排除某校/某专业）→ 只重跑 `Recommendation Agent → Risk Agent → Report Agent`，复用已有 `evidence_list` / `rule_results`，不重新检索、不重新跑规则校验，控制成本和延迟（预计 5-10 秒量级，不是完整 45 秒流程）。
   - **重大约束**（省份、选科、批次变更）→ 提示"这类修改会影响证据检索范围，需要重新生成完整报告"，走完整 Agent 流程。
   - 返回新的 `report_version`（报告增加版本概念，`reports` 表新增 `parent_report_id` / `version` 字段，同一 `profile_id` 下的多个版本可追溯）。
4. 报告页顶部增加版本切换（`v1` / `v2` ...），聊天面板里对应给出反馈："已根据新预算重新生成，候选从 48 所变为 31 所，均衡型方案里新增 2 所"。

**技术风险**：需验证 LiteLLM → Moonshot Kimi k2.6 在**流式回复 + tool calling 同时开启**场景下的稳定性（现有 Agent 节点的工具调用都是非流式的同步调用，ConversationAgent 是本系统里第一个"流式 + 工具调用"并存的场景），建议 Phase 2d 启动前先做小范围技术验证（spike），不确定性高于本文档其他改造点。

**验收标准**：
- **Given** 用户在聊天面板输入"预算超过 10 万的别推荐了"，**When** AI 识别为约束修改并经用户确认，**Then** 系统在 10 秒内返回基于新约束重新过滤/排序后的方案，且明确说明变化了什么。
- **Given** 用户要求修改省份，**When** 触发重新生成确认，**Then** 系统明确告知"这类修改需要完整重新生成，预计耗时与首次生成一致"，避免用户误以为都是秒级响应。

### 3.3 方案对比模式（新增）

- 入口：报告页三套方案 Tab 旁新增「对比」按钮手动触发；同时支持对话触发——用户说"帮我对比均衡型和进取型"，ConversationAgent 直接调用 `open_compare_view` 工具打开同一个视图，两种入口共用一套组件。以及产生多个 `report_version` 后，版本切换器旁新增「对比其他版本」。
- 交互：勾选任意两个方案（可以是保守 vs 均衡，也可以是 v1 vs v2），弹出对比视图（Bottom Sheet / Drawer，与证据链抽屉一致的呈现方式）：逐项对比学校、专业、位次差、学费、风险等级，差异项高亮标出。
- 底部固定一句 AI 生成的取舍建议（复用 report-agent 模型，走已有 LiteLLM 网关，非新增模型依赖）：例如"方案 A 更稳但学费更高，方案 B 冲的成分更多但性价比更好，求稳建议选 A"。
- 验收标准：
  - **Given** 用户勾选两个方案进入对比，**When** 对比视图渲染完成，**Then** 至少展示学校/专业/位次差/学费/风险等级 5 个维度的逐项差异，且差异项有视觉区分（非纯文字罗列）。

### 3.4 结果呈现优化（竞品借鉴，来自千问高考分析）

两点低成本高价值补充，风险低、复用度高，可并入 Phase 2a/2b 一起做，不必等到需要架构改造的 2c/2d：

1. **生成后 AI 点评+收窄建议**：报告生成完成后，在考生概况卡片下方追加一段 AI 生成的"条件点评"，指出用户输入条件里的张力或可优化点（例如"你的地域偏好和预算存在冲突，如果放宽 XX，候选会更充分"）。这是"AI 主动澄清"思路成本最低的变体——不需要在建档阶段打断表单流程，事后一段话就能传达"AI 认真分析过你的输入，不是套模板"。复用 report-agent 模型生成，不新增架构。
2. **候选卡片数据密度提升**：当前推荐卡片主要信息是"安全度数字+进度条"，参考竞品补充**近两年历史投档位次并列展示** + **概率精确到百分比**（而非只有冲/稳/保三档定性标签），让"综合评分"从抽象数字变成可追溯的历史依据，不影响现有 Tab/卡片结构，只是卡片内信息升级（补充 frontend-prd.md §8.6 推荐卡片规范）。

## 4. 技术改造映射（复用 vs 新增，控制工作量）

| 能力 | 复用（无需新建） | 新增 |
| --- | --- | --- |
| Agent 协作时间线 | SSE 架构（`sse:{run_id}`）、Debug 事件的原始数据（`parallel_fan_out/in`、`reflection_iteration`、`degraded`） | 用户侧事件白名单扩展（2.3）、前端时间线组件、"技术事件 → 用户文案"映射表 |
| 决策过程回放卡片 | 时间线数据结构 | `agent_runs` 或 `reports` 新增 `run_summary_json`（用户可见摘要，区别于已有 `debug_summary_json`） |
| 建档问诊 AI 追问 | Profile Agent 追问机制、`profile_pending_questions` 字段、Policy Rule Agent 规则工具 | 前端对话式追问卡片组件、建档流程在表单和追问之间的状态机 |
| 报告对话改约束 | ConversationAgent 框架、SSE 流式回复、LiteLLM `report-agent` 模型 | 意图识别分类（`question` vs `refine_request`）、约束抽取为结构化 patch、`POST /reports/{id}/refine` 接口、局部重跑逻辑（复用 State 但跳过 retrieval/policy_rule 阶段）、`reports` 表 `version`/`parent_report_id` 字段 |
| 方案对比 | 报告数据结构、report-agent 模型 | 对比视图组件、AI 取舍建议的 prompt 模板 |

**明确不做**（避免范围蔓延）：
- 不引入新的 LLM 供应商或模型（仍走 LiteLLM → Moonshot Kimi）。
- 不对已完成 run 做"边跑边改约束"的实时打断（`refine` 只作用于已完成的报告，不是运行中的 run 支持 interrupt）。
- Admin Debug Console 保留现状，不因为本次改造而重构，两者共享事件源但呈现层独立演进。

## 5. 分期建议

| 阶段 | 内容 | 理由 |
| --- | --- | --- |
| Phase 2a | 生成进度页「协作时间线」+ Reflection 可视化（§2.2、2.3） | 改动集中在事件转译层 + 一个前端组件，复用度最高，见效最快，是"AI 全程可见"最直观的证明 |
| Phase 2b | 报告页「决策过程回放」卡片（§2.4） | 直接复用 2a 产生的数据，边际成本低 |
| Phase 2c | 建档问诊 AI 追问（§3.1） | 需要前端交互模式改造（表单 ↔ 对话混合状态机），比 2a/2b 复杂 |
| Phase 2d | 报告对话改约束重新生成（§3.2） | 涉及意图识别、局部重跑、版本管理，工作量最大，建议放最后 |
| Phase 2e | 方案对比（§3.3） | 依赖 2d 的版本概念（否则只能对比三套静态方案，价值打折） |
