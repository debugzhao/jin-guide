## 0. 产品定位与设计哲学

问津 Agent 不是"填表单生成报告的工具"，也不是"套壳大模型的纯聊天机器人"。它是一个**AI 全程协作者**：从建档、生成到报告交付的每一步，用户都能感知到 Agent 在做什么、为什么这么判断，并且可以随时用自然语言介入、调整、追问。

这个定位落地为一个具体的架构选择——**Generative UI 混合形态**：

- **Chat 是控制层**：用户表达意图（回答问题、修改约束、要求对比）的唯一入口是自然语言对话。
- **结构化视图是呈现层**：Tab、卡片、风险总览、对比视图、表单控件，是 Agent 用来回应用户、承载信息的画布。
- **两者不是割裂的两个目的地**，而是同一个 Agent 交互的两种输出形式。Agent 判断某个响应该用文字说清楚，还是该直接操作画布上的某个组件（切换 Tab、高亮卡片、弹出对比视图），取决于内容本身的性质，不取决于"现在是哪个页面"。

**为什么不做成纯聊天界面**：志愿填报是信息密集、需要反复横向比较的决策任务——几十条候选院校、冲稳保分层、位次差、风险标记，用户需要"扫一眼定位、随时回看对比"。聊天记录是线性的、一次性的，无法承载这种需求；市面上把报告直接挂进通用 Chat 对话的产品，实际操作性明显弱于结构化界面。所以问津选择让 Chat 驱动结构化组件，而不是用 Chat 取代结构化组件。

这个哲学在当前原型中收敛为一个核心工作台，而不是多个独立流程页——**AI 对话建档页和报告工作台本质是同一个左右双栏组件树的两个成熟度阶段，不是需要跳转切换的两个独立页面**：

| 页面 | 对话驱动的内容 | 结构化呈现的内容 |
| --- | --- | --- |
| AI 对话建档（§6.1） | Agent 连续收集省份、分数、位次、选科等必填信息，并在矛盾时追问；确认基础信息后，用户可继续用自然语言或结构化卡片补充预算、城市、专业偏好 | 每条 AI 消息可渲染下拉、数字输入、多选 chip、风险画像、档案摘要、生成过程卡片；**右侧实时报告面板**在最小必填信息确认后立即渲染"基础版"报告，并随偏好补充持续更新为"偏好更新版"——报告不是生成完成后才出现的独立步骤，而是从首次达到生成条件起就持续可见、持续演进 |
| 报告工作台（§6.2） | 左侧对话持续追问、解释、改约束、打开对比 | 右侧实时/最终渲染报告，包含考生概况、AI 点评、风险总览、决策回放、三套方案、证据链和对比抽屉——与 AI 对话建档页的实时报告面板是**同一套组件**，只是在报告拿到可分享的 `report_id` 后，前端把地址栏无刷新切换到 `/reports/[id]`，便于历史记录访问、书签和分享 |

### 设计原则

- **过程可见**：凡是 Agent 产生的协作信号（并行执行、自我修正、降级重试），只要不涉及技术黑话或用户 PII，就要有一个用户能看懂的呈现方式，不能只留在开发者可见的调试工具里。
- **对话优先于表单，但不迷信对话**：需要澄清歧义或收集有判断成分的信息时，用"AI 追问 + 用户回答"；纯粹结构化、无歧义的原子字段（省份、分数这类）仍然用结构化控件呈现，只是不再包在固定步骤的表单外壳里——控件本身也是 Agent 渲染出来的内容，只是不需要每次都经过 LLM 判断。
- **报告是起点不是终点**：报告生成完成不等于交互结束，用户应该能在报告基础上继续表达"改一下约束""这个和那个比比看"，AI 能听懂意图并采取行动，而不是只能回答"为什么推荐这个"。
- **技术真实性**：所有"看起来很智能"的呈现必须对应后端真实发生的事情，不能为了好看而编造不存在的 Agent 行为。
- 核心体验是"左侧对话 + 右侧报告画布"，桌面端三栏（侧栏 + 对话 + 报告）呈现最完整；`/`、`/reports/[id]` 已做响应式适配（见 §10），Admin Debug 控制台仍只做桌面端体验。
- 首屏直接进入 AI 对话建档，不做营销式落地页，也不做入口选择页。
- 快速测算、建档问诊、生成进度都不是独立页面，而是对话流中由 Agent 生成的结构化 UI 卡片。
- 报告不是"生成完成后才出现的终点物"：最小必填信息一确认，AI 对话建档页右侧就会实时渲染报告面板（基础版），并随对话中补充的偏好持续更新为偏好更新版；用户全程停留在同一页面，不会被强制跳转。
- 不使用"精准预测""必上""保录"等高风险文案。
- 所有推荐结论必须有证据入口。
- 高风险项要明显、可解释、有据可查。
- 不把基础风险藏起来，当前版本所有功能免费开放。
- 报告页面要适合桌面端长阅读、横向比较和面试演示。

## 1. 前端目标

前端的核心目标是让高考生和家长在 Web 端，通过一个持续在场的 AI 协作者完成：

- 默认进入 AI 对话建档，不经过入口页和营销页。
- 在同一条对话流里完成快速测算、档案收集、规则前置校验、偏好补充和志愿草稿体检。
- 点击生成后，在当前对话中看到 Agent 真实工作状态——并行检索/规则校验、自我检查、降级兜底——不跳转独立生成进度页。
- 生成最终志愿报告后，进入桌面端工作台：左侧持续对话，右侧结构化报告实时/最终渲染。
- 查看风险画像、三套冲稳保方案和证据链，并能继续追问、修改约束、要求方案对比。
- 在高风险场景中获得明确的风险说明，自行判断决策。

当前版本不承担商业化转化目标，前端不展示价格、套餐卡、订单状态、支付弹窗或付费墙。

## 2. 技术选型

### 2.1 推荐方案

首期使用 **Next.js + React + TypeScript**。

原因：

- 当前目标是网页版产品，不是原生小程序多端应用。
- Next.js 适合构建完整 Web 产品，内置路由、服务端渲染、数据获取、Route Handler、图片优化和部署生态。
- 本产品需要报告分享页、登录态、服务端数据预取和轻量 BFF，Next.js 的工程结构更完整。
- SSE 长连接（生成进度、协作时间线、报告问答流式回复）是产品的核心交互载体，Next.js Route Handler 可以承担轻量转发/鉴权职责。

### 2.2 建议技术栈

| 能力       | 推荐                                         |
| ---------- | -------------------------------------------- |
| 框架       | Next.js App Router                           |

## 3. 设计系统

### 3.1 设计基调

产品面向高考生和家长，这是一个高风险、高信息密度的决策场景。设计基调直接采用 `docs/wenjin-agent-prototype.html` 已验证的**纯白背景浅色风格**：少元素、强中心、留白大，用蓝色主色和克制的中性灰阶传递专业感，不做深色/玻璃拟态实验。设计基调应该是：

- **专业可信**：不是活泼的学生 App，也不是冰冷的政府网站，接近"可信赖的专业工具"；纯白背景 + 深蓝主色传递的是稳健和确定性。
- **克制大气**：白底上用少量留白和边框分层，不靠阴影堆叠制造层次；模块之间留白宽裕，不堆砌卡片，一屏只讲一个重点。
- **清晰易读**：信息密度高，但层次分明，关键数字和风险项要突出；白底上深色文字保证最高可读性，不牺牲对比度换视觉效果。
- **风险可视**：高/中/低风险必须靠颜色直观区分，用户无需阅读文字就能感知危险程度，具体色值见 §3.2。
- **对话自然**：Agent 的对话式交互要克制、专业，不使用拟人化的寒暄或表情堆砌，语气接近一个靠谱的顾问，而不是聊天软件里的虚拟角色。

### 3.2 色彩系统

#### 背景与层次

页面统一纯白背景，卡片用细边框 + 极弱阴影分层，不引入深色玻璃拟态。

| 层级 | 色值 | 说明 |
| --- | --- | --- |
| 页面基色 | `#FFFFFF` | 纯白，全站统一背景 |
| 次级背景 | `#F8FAFC` | 嵌套区块/输入框/Tab 未选中态的浅灰背景，制造弱层次而不引入阴影 |
| 卡片背景 | `#FFFFFF` | 卡片、面板、导航容器背景 |
| 卡片边框 | `#E2E8F0` | 细边框承担分层，阴影仅作为极弱的辅助（见 §3.5） |
| 弹层/Modal/Drawer 背景 | `#FFFFFF` + 遮罩 `rgba(15, 23, 42, 0.4)` | 登录 Modal、证据链抽屉、对比抽屉背后统一用深色半透明遮罩 |

#### 品牌色

| 用途   | 色值      | 说明                                               |
| ------ | --------- | -------------------------------------------------- |
| 主色   | `#1E40AF` | 深蓝，用于主按钮、链接、激活态、标题强调词，与 `wenjin-agent-prototype.html` 的 `--blue` 一致 |
| 辅助色 | `#0D9488` | 深青绿，用于进度条、安全度数值、正向指引 |

#### 风险等级色

这是整个产品最重要的视觉语言，必须全局统一。红/橙/绿/蓝四种色相语义不变，白底上直接用饱和文字色 + 极浅背景色，保证扫视时能第一时间识别高风险项。

| 高风险        | `#DC2626`   | `#FEF2F2`   | `#FECACA`   | 保底不足、选科冲突、体检限制命中，需要用户重点关注并调整的项 |

| 占位/禁用     | `#94A3B8`   |
| 分割线/边框   | `#E2E8F0`   |

### 3.3 字体与字号

字体优先使用系统字体，保证在主流桌面和移动浏览器上的渲染效果。

```
font-family: -apple-system, "PingFang SC", "Noto Sans SC", "Helvetica Neue", sans-serif
```

| 层级     | 字号 | 行高 | 字重 | 使用场景                     |
| -------- | ---- | ---- | ---- | ---------------------------- |
| 大标题   | 20px | 28px | 600  | 页面主标题                   |
| 小标题   | 17px | 24px | 600  | 卡片标题、章节标题           |

### 3.4 间距系统

基于 4px 基准：4 / 8 / 12 / 16 / 20 / 24 / 32 / 48px。整体遵循"大面积留白"原则——模块之间距离宁可大，也不把卡片塞满。

- 卡片内边距：16px
- 页面水平边距：16px
- 列表项间距：12px
- 章节间距：32px，让每屏只突出一个重点

### 3.5 圆角与阴影

- 卡片圆角：12px
- 按钮圆角：8px
- 标签/徽章圆角：4px
- 对话气泡圆角：16px（比卡片更圆润，视觉上区分"对话"和"数据卡片"两种内容类型）
- 卡片边框：`1px solid #E2E8F0`（细边框承担分层作用，阴影只是弱辅助）
- 卡片阴影：`0 1px 3px rgba(15, 23, 42, 0.08)`（极弱阴影，不做深色主题下那种厚重投影）

## 4. 信息架构

### 4.1 页面层级

```mermaid
flowchart TD
  A["AI 对话建档 /"] --> A1["快速测算卡片"]
  A --> A2["结构化建档控件"]
  A --> A3["风险画像卡片"]
  A --> A4["志愿草稿体检卡片（可选）"]
  A --> A5["对话内生成过程卡片（可展开，不阻塞报告展示）"]
  A --> A6["实时报告面板：空状态 → 基础版 → 偏好更新版（同页渲染，可折叠）"]
  A6 -. "拿到 report_id 后地址栏无刷新切换" .-> B["报告工作台 /reports/[id]"]
  B --> B1["左侧持续对话"]
  B --> B2["右侧报告画布（与 A6 同组件）"]
  B2 --> B3["证据链抽屉"]
  B2 --> B4["方案/版本对比抽屉"]
  B3 --> C["数据源详情 /sources/[id]"]
  B --> D["报告历史 /reports"]
  B --> E["Admin Debug /admin/debug"]
```

### 4.2 导航结构

当前版本是**桌面端 Agent 工作台**，用户不需要先选择入口，也不需要在多个流程页之间跳转。导航策略如下：

**左侧持久导航栏**（`≥ lg` 常驻，`< lg` 收进抽屉，见 §6.1「响应式断点」）：Logo + "新建对话" + 历史记录（当前版本为占位骨架，接入真实历史列表是后续迭代范围）+ 用户菜单/登录入口（底部）。

**不设置底部 Tab 导航栏**：产品是线性决策流程，底部 Tab 会破坏流程感，且当前功能模块不够独立。

**移动端/平板端**：`< lg`（1024px）断点下侧栏与报告栏分别收进抽屉和 BottomSheet，不是简单隐藏不可访问；当前版本交付响应式布局，不再是纯桌面端原型。

**报告历史入口**：登录后从侧栏历史记录进入（占位骨架版本暂无跳转）；未登录用户只保留匿名草稿和当前会话。

**关键跳转规则**：

- 首屏路由 `/` 即 AI 对话建档；快速测算、建档问诊、生成进度均在该页面以对话卡片出现。
- 必填字段（学生基础信息、选科、性别/体检限制）确认后即满足生成条件，不需要等预算/城市/专业等"建议"字段填完；点击"确认基础信息并渲染报告"后，不跳转 `/reports/generating`；前端在当前对话流追加生成过程卡片，并通过 SSE 更新状态。
- 首次生成完成后**不发生页面跳转或状态丢失**：报告直接渲染在 `/` 页面右侧的实时报告面板（"基础版"），左侧对话区继续承接后续追问和改约束；用户此时补充城市/专业/学费等偏好，面板经 `/refine` 更新为"偏好更新版"。
- 报告一旦拿到可分享的 `report_id`，前端把地址栏**无刷新**切换为 `/reports/[id]`——这只是同一份组件树的路由标注变化，不是导航到另一个页面；用户从报告历史、分享链接直接打开 `/reports/[id]` 时，加载的是同一套左右双栏工作台，直接以已生成报告的状态呈现。
- 右侧实时报告面板/报告画布支持折叠收起（收起时对话区占满宽度），悬浮的"展开报告"入口可随时唤回。
- 报告页的"AI 是如何得出这份方案的"卡片可回放生成过程摘要，不依赖独立生成进度页，也不需要先展开才能看到报告本身。

## 5. 核心流程

### 5.1 默认流程：对话建档 → 实时报告面板 → 偏好持续细化

```mermaid
flowchart TD
  A["用户进入 /"] --> B["AI 对话建档：快速测算 + 选科/体检等必填字段"]
  B --> C["确认基础信息"]
  C --> D["对话内生成过程卡片（可展开，不阻塞展示）"]
  D --> E["右侧实时报告面板：基础版报告"]
  E --> F["用户继续对话补充城市/专业/学费偏好"]
  F --> G["偏好确认卡片 → /refine 局部重新生成"]
  G --> E
  E --> H["拿到 report_id，地址栏无刷新切换为 /reports/[id]"]
  H --> I["报告工作台：左侧对话 + 右侧报告（同一组件）"]
  I --> J["持续追问 / 改约束 / 方案对比 / 版本切换"]
  J --> I
```

**关键规则**：

- 首屏即进入对话建档，不展示入口页。
- 快速测算不是页面，是第一组 AI 结构化卡片；用户也可以直接在底部输入框表达"我是河南 612 分，位次 32680"。
- 建档问诊不是页面，是连续对话流；字段由 Agent/规则驱动逐步出现。
- **必填字段一确认即触发首次生成**，不需要等预算/城市/专业等建议字段：报告直接以"基础版"渲染在同一页面右侧的实时报告面板，不是走完完整表单才看到报告。
- 生成过程不是独立页面，是对话内过程卡片；默认展示精简摘要，用户可展开查看完整并行/自检/降级时间线，但**不阻塞报告面板同时渲染**。
- 之后每补充一次城市/专业/学费等偏好，经确认后调用 `/refine` 更新同一份报告为"偏好更新版"，仍在同一页面，不产生新的报告实体或新页面。
- 报告拿到可分享的 `report_id` 后，地址栏无刷新切换为 `/reports/[id]`；这是同一套双栏组件的路由标注变化，不是导航到另一个页面，也保留了直接分享/收藏/历史记录访问的能力。

## 6. 页面需求

### 6.1 AI 对话建档页（`/`）

AI 对话建档页是当前版本的默认首页。它不是聊天框套表单，而是一个 **Generative UI 对话流**：AI 消息可以渲染文本、结构化输入控件、风险画像、规则校验结果、档案摘要和生成过程卡片。**页面进入时是纯聊天入口（Chat-first）**，只有识别到"开始建档/生成报告"意图后才在对话流里内联渲染建档表单，不是一进门就摊开表单；与 §6.2 报告工作台共享同一布局骨架和组件——区别只是右侧报告栏在纯聊天阶段不出现（避免空状态卡片过早撑开宽屏布局），随对话推进逐步出现并演进为"基础版报告"和"偏好更新版报告"。

#### Chat-first 建档入口

- 首屏是空态欢迎语（"你好，我是问津"）+ 建议 prompt（"开始志愿建档""帮我测算一下我的位次""解读一下一分一段表是什么"）+ 底部输入框，不预置任何表单。
- 用户发送消息后，前端调用 `POST /api/v1/profile/intent`（入参 `{message}`，出参 `{intent: "start_profile" | "chitchat"}`）判定是否命中建档意图；该接口内部先走 `profile-agent` 模型分类，LLM 超时/报错/输出格式不合法时降级到确定性关键词兜底（"建档""报志愿""生成报告""测算"等），接口恒可用、不阻塞对话。
- 命中 `start_profile`（或用户直接点"开始志愿建档"CTA，跳过模型调用）后，对话流内联渲染下方的「结构化建档卡片」，聊天历史保留在其上方，形成一条连续对话，而不是切换到另一个页面。
- 命中 `chitchat` 时，给出一句兜底引导文案 + 常驻的"直接开始志愿建档"入口，不生成报告、不渲染右侧报告栏。

#### 对话内卡片类型

| 卡片 | 触发时机 | 内容 | 后端接口 |
| --- | --- | --- | --- |
| Chat-first 意图判定 | 纯聊天阶段用户每次发送消息 | 判定 `start_profile`/`chitchat`，命中前者才继续渲染建档卡片 | `POST /api/v1/profile/intent` |
| 快速测算卡片 | 首次进入或用户表达"先测一下" | 省份、批次、分数、位次；提交后生成风险画像 | `POST /api/v1/risk/preview` |
| 结构化建档卡片 | 风险画像后连续出现 | 选科、性别/体检限制、预算、城市偏好、专业偏好、风险风格 | `POST /api/v1/profile`、`POST /api/v1/profile/field-check` |
| 规则前置校验卡片 | 字段提交后命中规则/歧义 | 例如选科、体检、批次冲突；提供"调整"或"仍按此继续"按钮 | `POST /api/v1/profile/field-check` |
| 志愿草稿体检卡片 | 用户说明已有志愿表 | 手动录入志愿草稿，展示风险项；不支持上传/OCR | `POST /api/v1/volunteer/check` |
| 基础信息确认卡片 | 学生基础信息、选科、性别/体检限制等必填字段填完 | 汇总必填字段，主按钮"确认基础信息并渲染报告"；不等待预算/城市/专业等建议字段 | 本地状态 + profile |
| 生成过程卡片 | 确认基础信息后自动触发首次生成；此后每次偏好确认触发局部重新生成 | 档案检查、数据版本锁定、并行处理、自我检查、降级提示、完成状态；默认展示一条精简摘要，可展开完整时间线，**不阻塞右侧实时报告面板同时渲染** | 首次：`POST /api/v1/reports/generate`；偏好更新：`POST /api/v1/reports/{id}/refine`，均配合 SSE |
| 推荐偏好卡片 | 用户在输入框自由表达城市/专业/学费等偏好，或点选结构化偏好 chip | 城市偏好、专业偏好、年学费上限；标注"按需生成"，非必填，确认后作为 `refine` 的 `patch` | 本地状态，确认后调用 `POST /api/v1/reports/{id}/refine` |

**实时报告面板**（右侧，贯穿建档到报告成熟的全过程，不是独立卡片）：

| 状态 | 触发条件 | 内容 |
| --- | --- | --- |
| 空状态 | 必填字段未填完 | "等待基础建档信息"提示卡 |
| 基础版 | 必填字段确认，首次生成完成 | 报告标题/生成时间、考生概况、AI 条件点评（如"目前只有基础建档信息，报告会先覆盖更多候选，避免过早收窄"）、风险总览、三套方案 Tab、候选卡片；标签"基础版" |
| 偏好更新版 | 偏好卡片确认后，`refine` 完成 | 同一份报告内容更新（候选收敛、点评更新为"已把城市/专业/学费偏好纳入排序"）；标签"偏好更新版" |

面板可通过顶部"收起报告"/悬浮"展开报告"按钮折叠或唤回，折叠时对话区占满宽度；折叠状态不清空已渲染的报告数据。

| ---- | ---- | -------------- | ---- |

| 学生基础信息 | 选科 | 物理/历史单选 + 其余科目多选 chip | 是 |

#### 确定性逻辑与 Agent 边界

| 环节 | 实现方式 | 是否调用 LLM |
| --- | --- | --- |
| Chat-first 建档意图判定 | `POST /api/v1/profile/intent`：`profile-agent` 模型分类 + 关键词兜底 | 是（失败时降级为否） |
| 下一个字段该问什么、是否跳过 | 确定性字段依赖图 | 否 |
| 单字段矛盾/歧义检测 | `POST /api/v1/profile/field-check` + Rule Engine | 否 |
| 把规则结果转成自然语言追问、理解自由文本回应 | Profile Agent | 是 |
| 快速风险画像 | Risk Engine 同步计算 | 否 |
| 生成过程状态 | LangGraph/ARQ 真实事件经 SSE 转译 | 否，事件来自真实执行 |

#### 交互规则

- 页面底部始终有输入框，用户可以随时补充"我想去上海""预算 8 万以内""不考虑护理"。
- AI 可以在回复中直接插入控件；用户点选控件与输入自然语言都写入同一份 profile state。
- 必填字段（学生基础信息、选科、性别/体检限制）未满足时，"确认基础信息并渲染报告"禁用，并提示缺失项；不要求预算/城市/专业等建议字段填完。
- 建档过程自动保存草稿；刷新后恢复对话历史、已填写字段和右侧实时报告面板当前状态（空/基础版/偏好更新版）。
- 首次生成和后续偏好更新都不跳页：SSE 事件在当前对话流中追加或更新同一张生成过程卡片，同时驱动右侧实时报告面板从空状态 → 基础版 → 偏好更新版演进。
- 报告拿到 `report_id` 后，地址栏无刷新切换为 `/reports/[id]`；用户可随时点击"在独立页面查看完整报告"主动触发，也可以留在 `/` 继续对话，两者渲染的是同一份数据。

#### 对话内生成过程卡片

展示 Agent 工作过程，但不作为独立页面，也不阻塞右侧报告面板的展示：

| 展示元素 | 触发事件 | 说明 |
| --- | --- | --- |
| 档案检查 / 数据版本锁定 | `node_started` / `node_completed` | 普通步骤 |
| 并行处理分组 | `agents_parallel_started` → `agents_parallel_merged` | 同时展示检索招生数据与规则校验 |
| 候选方案生成 | `candidates_ready` | 展示候选数量和冲稳保数量 |
| 风险发现 | `risk_found` | 展示高/中/低风险摘要 |
| 自我检查分组 | `self_check_round` | 只展示类别化原因，不展示原始违规文本 |
| 降级提示 | `degraded_notice` | 转译成安心文案，不暴露技术服务名 |
| 完成 | `completed` | 右侧实时报告面板由空状态/上一版本切换为最新报告（基础版或偏好更新版） |

### 6.2 报告工作台页（`/reports/[id]`）

报告页是产品核心交付物，也是当前作品集的主展示页面。它采用桌面端双栏布局：**左侧持续对话，右侧结构化报告画布**。这部分对齐 `wenjin-agent-prototype.html` 的千问式交互，但报告内容保留问津的证据链、风险和决策回放深度。

**与 §6.1 实时报告面板的关系**：`/reports/[id]` 不是用户点击"生成"后被动跳转到的另一个页面，而是 §6.1 实时报告面板在拿到可分享 `report_id` 后的独立路由呈现——两者是同一套左右双栏组件（`ReportWorkspaceLayout`/`PlanTabs`/`CandidateCard`/`RiskOverview`/`EvidenceDrawer`/`PlanCompareView`），差异只在于：`/` 页面同时承担建档职责（结构化控件、澄清追问），`/reports/[id]` 默认以文档式排版突出报告阅读体验，并作为报告历史、分享链接的固定入口。以下描述的所有交互（改约束、方案对比、版本切换）在 `/` 页面报告面板生成基础版之后同样可用，不需要先跳到 `/reports/[id]`。

#### 整体布局（桌面双栏）

```
┌────────────────────────────────────────────────────────────────────┐
│ 顶部栏：返回 / 完整报告 / 登录                                     │
├──────────────────────────────┬─────────────────────────────────────┤
│ 左侧对话栏                    │ 右侧报告画布                         │
│  - 历史消息                   │  - 报告标题/生成时间                  │
│  - 生成过程入口               │  - 考生概况                           │
│  - 预设追问                   │  - AI 条件点评                         │
│  - 改约束确认卡片             │  - 风险总览                           │
│  - 底部输入框                 │  - AI 是如何得出这份方案的             │
│                               │  - 三套方案 Tab + 推荐卡片             │
│                               │  - 证据链抽屉 / 方案对比抽屉           │
└──────────────────────────────┴─────────────────────────────────────┘
```

#### 左侧对话栏

- 固定在报告页左侧，不使用右下角悬浮按钮，不使用 Chat Panel 抽屉。
- 输入框固定在左栏底部，支持 Enter 发送。
- 预设问题：
  - "推荐的学校是怎么筛选出来的？"
  - "帮我对比均衡型和进取型"
  - "预算超过 8 万的别推荐了"
- 用户可以继续提问、打开对比、修改约束；AI 回复以气泡展示，工具执行反馈使用浅绿背景。
- 数据变更类意图必须展示确认卡片，用户确认后才调用 `/reports/{id}/refine`。

#### 右侧报告画布

右侧报告画布以文档式长阅读为主，但关键模块仍是可交互组件：

#### 三套方案 Tab

- 默认展示"均衡型"Tab。
- Tab 下方滚动展示该方案的推荐卡片列表（3-5 张，按冲→稳→保顺序排列）。
- 切换 Tab 时列表滚动到顶部，不保留滚动位置。
- Tab 可由用户手动点击切换，也可由 ConversationAgent 通过 `switch_tab(tier)` 工具直接切换，两种触发方式共用同一套状态。

#### 推荐卡片

每张卡片展示以下信息：

**主要信息（始终可见）**：

- 学校名称（大字）+ 城市 + 层级标签（冲/稳/保/高冲，色彩区分）
- 专业名称 + 专业组代码
- 模拟录取安全度（数字 + 进度条，颜色与安全等级对应）+ **匹配置信分**（如 `78/100`，用于排序解释，不表达确定性录取概率）
- 综合评分 + 学费/年

**次要信息（点击展开）**：

- 选科要求
- **近两年历史投档位次并列展示**（如 `2024: 38500  2025: 37200`），让"综合评分"从抽象数字变成可追溯的历史依据
- 推荐理由（2-3 条结构化说明）
- 风险提示（如有，红色/橙色小标签）
- 证据入口：点击"查看数据来源"打开证据链抽屉
- 支持勾选加入「方案对比」

#### 证据链抽屉

桌面端从右侧滑出，不全屏覆盖：

- 来源标题、数据年份、省份
- 权威级别标签（官方 / 高 / 中）
- 关键字段列表（招生计划数、最低位次、专业组代码等）
- 短引用摘要
- 底部：数据来源完整页入口

#### 风险总览卡片

- 整体安全等级（高/中/低，大色块）。
- 关键风险列表（最多展示 3 条，其余折叠）。
- 每条风险：等级图标 + 一句描述 + 来源志愿位置。
- "查看全部风险"展开按钮。
- 可由 ConversationAgent 通过 `expand_risk_detail(id)` 工具直接展开某条风险详情。

#### 生成后 AI 点评卡片

- 位置：考生概况卡片下方。
- 内容：报告生成完成后，追加一段 AI 生成的"条件点评"，指出用户输入条件里的张力或可优化点，例如"你的地域偏好和预算存在冲突，如果放宽 XX，候选会更充分"。
- 生成方式：复用 report-agent 模型，作为报告生成流程的一部分产出，不新增额外 API 调用。
- 价值：不需要在建档阶段打断表单流程，事后一段话就能传达"AI 认真分析过你的输入，不是套模板"。

#### 「AI 是如何得出这份方案的」决策过程回放卡片

- 位置：风险总览卡片下方，默认折叠。
- 内容：复用生成过程中记录的时间线数据（`run_summary_json`），**只读回放**，不重新调用 Agent。展示节选：并行处理过哪些任务、Reflection 修正了几轮、是否发生过降级——即 AI 对话建档页中"生成过程卡片"的精简回放版。
- 价值：把生成过程中转瞬即逝的信息沉淀成报告的一部分，用户事后想验证"这份报告靠不靠谱"时可以回来看。

#### 方案对比视图

- **入口**：三套方案 Tab 旁的「对比」按钮（手动勾选任意两个方案触发）；版本切换器旁的「对比其他版本」；对话触发（用户说"帮我对比均衡型和进取型"，ConversationAgent 调用 `open_compare_view(target_a, target_b)` 工具打开同一个视图）。`target` 统一为 `{ type: "plan" | "candidate" | "version", id: string }`，三种入口共用一套组件。
- **呈现**：右侧 Drawer，与证据链抽屉一致的呈现方式，逐项对比学校、专业、位次差、学费、风险等级共 5 个维度，差异项高亮标出（非纯文字罗列）。
- **AI 取舍建议**：底部固定一句 AI 生成的取舍建议（复用 report-agent 模型），例如"方案 A 安全边际更高但学费更高，方案 B 目标层级更高但不确定性更大，求稳可优先看 A"。文案只做取舍解释，不替用户作最终填报决定。
- **验收标准**：勾选两个方案进入对比后，视图渲染完成时至少展示学校/专业/位次差/学费/风险等级 5 个维度的逐项差异，且差异项有视觉区分。

#### 报告对话意图处理

ConversationAgent 收到左侧对话栏消息后判断走哪条路径：

| 意图类型 | 触发工具 | 前端表现 | 是否需要二次确认 |
| --- | --- | --- | --- |
| 答疑 | 无（纯文字回复） | 普通 AI 回复气泡 | 否 |
| UI 操作 | `switch_tab` / `highlight_candidates` / `open_compare_view` / `expand_risk_detail` | 画布对应组件立即联动（切 Tab / 高亮卡片 / 弹出对比视图 / 展开风险详情），AI 回复带一句确认文案（"已为你打开对比视图"） | 否，不涉及重新计算，可撤销、无成本 |
| 数据变更（改约束重新生成） | `regenerate_recommendations(patch)` | 聊天气泡下方展示确认卡片，用户确认后调用 `POST /api/v1/reports/{report_id}/refine` | 是，避免误触发重新生成消耗成本 |

数据变更类确认卡片样式：

```
检测到你想调整：预算上限 → 8 万元/年
[确认，重新生成方案]  [取消，继续问答]
```

确认后：

- **轻量约束**（预算、城市偏好、排除某校/某专业）：局部重新生成。用户确认后立即进入小型生成状态，预计 5-10 秒完成；完成后聊天面板给出反馈，例如"已根据新预算重新生成，候选从 48 所变为 31 所，均衡型方案里新增 2 所"。
- **重大约束**（省份、选科、批次变更）：AI 明确告知"这类修改需要完整重新生成，预计耗时与首次生成一致"，用户确认后回到 AI 对话建档页并在对话内展示完整生成过程卡片。
- 成功后报告页顶部出现/更新**版本切换器**（`v1` / `v2` ...），可在版本间切换查看。

**消息气泡规范**：

| 角色 | 对齐 | 背景色 | 说明 |
| --- | --- | --- | --- |
| 用户消息 | 右对齐 | `rgba(169, 180, 245, 0.12)`（提示色组浅底） | 纯文字，气泡尾朝右 |
| AI 回复 | 左对齐 | 玻璃卡片背景 `rgba(255, 255, 255, 0.04)` + 细白边 | 含引用标签，气泡尾朝左，顶部有"问津助手"小标签 |
| AI 工具执行反馈 | 左对齐 | 低风险色组 `rgba(143, 224, 183, 0.12)`（区别于普通回复） | 一句话确认文案，如"已为你打开对比视图" |
| 生成中状态 | 左对齐 | 玻璃卡片背景 `rgba(255, 255, 255, 0.04)` | 展示三点打字动画（`•••`），无文字 |

**行内引用标签**：

AI 回复中引用了具体数据来源时，在对应句子末尾展示行内标签（小圆角徽章）：

```
…2025年位次约 38500（来源：河南省2025招生计划 ↗）
```

- 标签点击跳转到 `/sources/[id]`（证据来源详情页，新标签页打开）。
- 桌面端悬停时展示 Tooltip（来源标题 + 年份 + 权威级别）。

**合规提示**：

- 若 ConversationAgent 触发合规修正（`compliance_warning` 事件），气泡底部展示灰色小字：「内容已通过安全审核」。
- 所有 AI 回复底部展示固定一行灰色说明文字（`13px`）：「以上分析基于报告数据，最终填报请以省级考试院为准」。

**操作控件**：

- 左侧底部固定输入区：文本框（最多 200 字，超出禁止输入）+ 右侧发送按钮（Enter 发送）。
- 左侧对话栏顶部可展示"清除对话"文字链，点击后二次确认弹窗，确认后清空历史并回到预设问题状态。
- 对话历史超过 40 条时，顶部展示提示条：「对话较长，建议清除后重新提问以保持上下文准确」。

**错误与限制状态**：

| 场景 | 处理 |
| --- | --- |
| 每日 30 条上限已用完 | 输入框禁用，展示「今日提问次数已达上限（30/30），明日重置」 |
| SSE 流中断 | 气泡底部展示「回复中断，点击重试」，点击后重发最后一条消息 |
| 网络错误 | Toast 提示「问答服务暂时不可用，请稍后再试」 |
| 对话历史已过期（7天） | 打开 Panel 时提示「上次对话已过期，已为你清除」，显示初始建议问题 |
| `/refine` 重新生成失败 | 确认卡片下方展示「重新生成失败，点击重试」，不影响当前已展示的报告版本 |

**验收标准**：

- 用户在聊天面板输入"预算超过 10 万的别推荐了"，AI 识别为约束修改并经用户确认后，系统立即展示局部重新生成状态，并在 10 秒左右完成基于新约束的过滤/排序，且明确说明变化了什么。
- 用户要求修改省份，触发重新生成确认时，系统明确告知"这类修改需要完整重新生成，预计耗时与首次生成一致"。
- UI 操作类工具（如 `open_compare_view`）执行时不展示确认卡片，直接联动画布并给出一句话反馈。

## 7. UI 状态规范

每个关键页面和组件必须定义以下状态，防止在开发过程中遗漏：

### 7.1 加载状态

- 推荐卡片列表：展示 3 张骨架屏卡片（Skeleton），高度与真实卡片相同。
- 证据链抽屉：内容区展示 2 行骨架屏文字。
- 报告工作台初次加载：左侧对话栏先渲染历史消息骨架，右侧考生概况和风险总览先渲染骨架，Tab 区域占位。
- 对话内生成过程卡片不使用骨架屏，用真实 SSE 事件传达进度。

### 7.2 空状态

| 场景             | 展示内容                                      |
| ---------------- | --------------------------------------------- |
| 报告历史无报告   | 插图 + "你还没有生成过报告" + 跳转 AI 对话建档按钮 |
| 志愿草稿未录入   | 对话内提示"可以手动输入已有志愿草稿，也可以跳过"     |
| 风险体检无风险   | 绿色图标 + "未发现明显风险，建议继续完整建档" |

### 7.3 React 错误边界

以下关键区域必须包裹 `ErrorBoundary`，避免单个组件崩溃白屏整页：

| 区域                     | 降级展示                                            |
| ------------------------ | --------------------------------------------------- |
| `CandidateCard` 列表渲染 | "部分推荐卡片加载失败，请刷新页面" + 重试按钮       |
| `EvidenceDrawer` 内容    | "证据详情暂时无法显示"                              |
| `AgentCollaborationTimeline` | 仅展示静态"生成中..."文案，不显示动态步骤       |
| 整个 `/reports/[id]` 页  | 友好错误页 + "返回首页"按钮，不展示任何原始错误堆栈 |

### 7.4 错误状态

| 场景                | 处理方式                                                                               |
| ------------------- | ---------------------------------------------------------------------------------------- |
| 网络请求失败        | Toast 提示"网络异常，请重试" + 请求重试按钮（不清空已填内容）                          |
| Agent 超时（> 60s） | 对话内生成过程卡片提示"生成时间较长，可稍后回来查看"；用户重新进入后通过 run_id 恢复状态 |
| 省份数据不覆盖      | 信息蓝色提示条，说明当前省份数据有限，建议补充位次以提高准确性                         |
| SSE 连接断开        | 自动重连（最多 3 次），3 次失败后展示"连接中断，点击刷新"按钮                          |

### 7.5 禁用/不可操作状态

- 档案完整度低于 40% 时，"生成三套方案"按钮显示禁用态，Tooltip 提示"请先完成基础信息填写"。
- 报告生成中时，所有修改档案的入口禁用。
- 数据版本未发布时，报告详情页展示"当前报告依赖的数据版本未完成校验"提示条。

## 9. 状态管理

### 9.1 本地状态（Zustand）

- 当前工作台模式（`conversation` / `report`）。
- AI 对话建档历史（已回答字段、结构化卡片、生成过程卡片）+ 档案完整度。
- 表单临时状态（未提交的字段值）。
- 志愿草稿（对话内可选卡片）。
- 实时报告面板状态 `liveReportStage: 'empty' | 'basic' | 'refined'`，与已生成的 `report_id`/`report_version` 关联；刷新页面后按此状态恢复面板展示，不回退到空状态。
- 实时报告面板折叠/展开开关（`isLiveReportOpen: boolean`，默认 `true`，仅在 `liveReportStage !== 'empty'` 时可展开）。
- 报告页当前 Tab（保守/均衡/进取），支持 Agent 工具 `switch_tab` 写入。
- 报告页当前选中的 `report_version`（默认最新版本）。
- 方案对比视图选中的两个对比对象 + 展开/收起状态。
- 证据链 Drawer / 方案对比 Drawer 展开状态。
- 当前对话 `conversation_id`（首次收到服务端 `done` 事件后写入，用于后续消息关联同一对话）。
- 对话消息列表（前端展示用缓存，`ChatMessage[]`，含流式生成中的临时消息）。
- 正在生成中的消息流式 token 缓冲（`streamingToken: string`，每个 `token` 事件追加）。
- **Debug 状态（admin 专属，非 admin 用户不初始化）**：
  - 当前选中的 `selectedRunId`。
  - "实时跟随"模式开关（`isLiveFollowing: boolean`）。
  - Run 列表筛选条件（`debugRunFilter: {status, degraded_only}`）。
  - 当前 run 的节点状态 Map（`nodeStates: Record<NodeName, NodeStatus>`，由 Debug SSE 事件驱动更新）。
  - 当前 run 的事件时间线（`debugEvents: DebugEvent[]`）。
  - 事件时间线筛选（`timelineFilter: 'all' | 'nodes' | 'tools' | 'errors'`）。
  - Auto-scroll 开关（`isAutoScroll: boolean`，默认 true）。

### 9.2 服务端状态（TanStack Query）

- 用户会话和登录态。
- 学生档案（含完整度）。
- 报告详情和版本列表。
- Agent run 状态（SSE 补充）。
- 志愿表体检结果。
- 数据源详情。

TanStack Query 配置要求：

- 报告详情：`staleTime: 30s`，用户主动刷新时重新请求。
- `POST /reports/{id}/refine` 返回 `202 + run_id` 后，前端订阅局部重新生成 SSE；收到 `completed` 事件后，`invalidateQueries` 使该 `profile_id` 下的报告版本列表失效并重新拉取，前端切换到新返回的 `report_version`。
- Agent run 状态：通过 SSE 更新，不使用轮询。
- 档案数据：`staleTime: Infinity`，明确 mutation 后失效。
- mutation 成功后统一调用 `queryClient.invalidateQueries` 失效相关 key。

### 9.3 SSE 鉴权方案

`EventSource` API 不支持自定义请求头，无法直接传 Bearer Token。前端统一使用以下方案：

- **Cookie 鉴权**（首选）：登录后由 BFF 种下 `HttpOnly` Session Cookie，SSE 连接自动携带，BFF 层校验后转发到 FastAPI。前端无需额外处理。
- **Query Token 备选**：若无法使用 Cookie（跨域等），先调用 `POST /api/v1/agent/runs/{id}/stream-token` 获取一次性短期 token（60s 有效），再拼入 SSE URL：`/api/v1/agent/runs/{id}/events?token=xxx`。

### 9.4 站内通知机制

报告生成完成、run 失败等异步事件需通知用户。当前版本不引入 Web Push（需 Service Worker，复杂度高），采用**长连接 SSE 推送**：

- 用户在任意页面时，前端建立 `GET /api/v1/notifications/stream`（永久 SSE 连接）。
- 后端有新通知时推送 `event: notification` 事件。
- 前端收到后在顶部展示 Toast 并刷新通知列表。
- 用户关闭页面后 SSE 断开，重新进入时调用 `GET /api/v1/notifications?unread=true` 补全遗漏通知。

## 10. 响应式与桌面端兼容性要求

### 10.1 布局断点（`/`、`/reports/[id]`）

| 宽度范围 | 布局策略 |
| --- | --- |
| < 1024px（手机/平板） | 侧栏收进抽屉（顶部汉堡呼出），主区单栏，报告栏收进悬浮"查看报告"按钮 + 底部 BottomSheet |
| ≥ 1024px（桌面） | 三栏：侧栏固定宽（~260px）常驻 + 对话列（限阅读宽度居中）+ 报告列（命中建档意图后出现，可折叠） |
| ≥ 1920px（超宽屏） | 整体壳体设总宽上限，避免三栏继续无限拉伸产生新的大片留白 |

**Admin Debug 控制台（`/admin/debug`）例外**：作为内部调试工具不做响应式适配，< 1024px 仍展示"请在桌面端使用 Debug 控制台"提示（见 §6.9）。

- 主要测试基准：390px（手机）、900px（平板）、1440px、1920px（桌面/超宽屏）。
- 报告画布内部长内容纵向滚动，不出现页面级横向滚动。
- 表单控件不溢出，Select 和多选使用自定义组件保证跨浏览器一致性。

### 10.2 交互适配

- 桌面端：支持 hover 状态提示，键盘焦点可见（`focus-visible`）。
- 对话输入框、报告左侧对话栏、右侧报告画布均需支持键盘滚动和焦点管理。
- Drawer 打开时，焦点留在 Drawer 内；关闭后焦点回到触发按钮。

### 10.3 分享页 OG 标签

报告分享页（`/reports/[id]`）需设置 OG 标签，保证在社交平台分享时展示正确的预览信息：

- `og:title`：`[学生名] 的问津志愿规划报告`
- `og:description`：省份、分数/位次、整体安全等级摘要
- `og:image`：报告封面缩略图（Next.js OG image 动态生成）

## 11. 文案与合规

### 11.1 禁止文案

在任何页面、按钮、标签、提示文字中禁止出现：

- 保录、必中、精准录取、包过、保上。
- 内部数据、独家数据、独家渠道。
- 付费后更容易录取、付费解锁完整报告。
- 任何形式的录取概率精确承诺（"录取率 85%"等）。

### 11.2 推荐文案

| 场景         | 推荐文案                                           |
| ------------ | --------------------------------------------------- |
| 风险评估结论 | "基于当前数据的模拟风险评估"                       |
| 推荐理由     | "该结论依赖 [数据年份] [省份] 数据版本"            |
| 有风险项时   | "请结合自身情况重点关注，最终填报以省级考试院为准" |
| 证据来源     | "来源：[来源名称]，[年份]，权威级别：[官方/高/中]" |
| 无法确定时   | "当前数据覆盖有限，以下为估算结果，仅供参考"       |

## 12. 前端验收标准

### 12.1 功能验收

- 默认进入 `/` AI 对话建档，不展示入口页、快速测算页、建档问诊页、生成进度页和移动端切换。
- 匿名用户可以完成测算和建档，登录后数据自动绑定。
- 快速测算以对话内结构化卡片出现，提交后在当前对话流展示即时风险画像。
- 建档问诊以对话流 + 结构化控件完成，不展示固定的"步骤 X/N"，草稿在刷新后可恢复。
- 志愿草稿体检以对话内可选卡片出现，不支持上传、OCR 或自动解析。
- 必填字段（学生基础信息、选科、性别/体检限制）确认后即可点击"确认基础信息并渲染报告"，不要求预算/城市/专业等建议字段填完；点击后在当前对话中展示生成过程卡片，SSE 事件实时更新该卡片，不跳转独立生成进度页。
- 对话内生成过程卡片收到 `agents_parallel_started` 后 1 秒内渲染"并行处理"分组；收到 `agents_parallel_merged` 后保留结果摘要。
- 对话内生成过程卡片收到 `self_check_round` 时只展示类别化修正原因；收到 `degraded_notice` 时展示转译后的安心文案，不暴露技术服务名。
- 首次生成完成后，右侧实时报告面板在 `/` 页面原地渲染"基础版"报告（考生概况、AI 点评、风险总览、三套方案），**不发生页面跳转**；用户可继续在同一页面对话补充偏好。
- 用户确认偏好卡片后触发 `/refine`，成功后实时报告面板更新为"偏好更新版"，标签同步变化，报告内容不重置为空。
- 报告拿到 `report_id` 后地址栏无刷新切换为 `/reports/[id]`；直接打开该 URL（历史记录/分享链接）能加载出同一份已生成报告，而非要求重新走一遍建档流程。
- 实时报告面板/报告画布支持折叠收起（"收起报告"）和悬浮"展开报告"唤回，折叠不清空已生成的报告数据。
- 报告工作台固定为桌面双栏：左侧常驻对话栏，右侧报告画布；不展示右下角"问一问"浮动按钮和 Chat Panel 抽屉。
- 报告页三套方案 Tab 切换正常，推荐卡片展开/收起正常。
- 证据链抽屉可以从推荐卡片唤起，展示正确的来源信息。
- 高风险报告展示明确的风险说明，用户自行判断决策。
- 报告历史页展示过往报告列表。
- 志愿草稿录入最多条数与当前省份上限一致（从 data/availability 接口读取），不硬编码 30 条。
- 关键渲染区域有 ErrorBoundary，单组件崩溃不影响其他区域展示。
- 左侧对话栏预设建议问题点击后立即发送并消失，历史对话不重复展示建议问题。
- AI 回复中的引用标签（`CitationInline`）点击跳转到 `/sources/[id]`（新标签页）。
- 每日 30 条上限触达后输入框禁用，并展示明确的上限提示文字。
- AI 所有回复底部均展示「以上分析基于报告数据，最终填报请以省级考试院为准」。
- 合规修正事件（`compliance_warning`）触发后，对应气泡展示「内容已通过安全审核」小字。
- 选科组合在目标省份/批次无历史招生数据时，AI 对话建档在提交该字段后立即展示澄清追问气泡，而非等全部字段填完后统一提示。
- 报告页「AI 是如何得出这份方案的」卡片默认折叠，展开后只读回放生成过程时间线，不触发任何 Agent 调用。
- 推荐卡片展示匹配置信分和近两年历史投档位次并列数据，不再只有冲/稳/保定性标签；页面不出现"录取概率 X%"这类确定性概率表达。
- 左侧对话栏识别到 UI 操作类意图（`switch_tab`/`highlight_candidates`/`open_compare_view`/`expand_risk_detail`）时直接执行并联动画布，不展示确认卡片；识别到数据变更类意图（`regenerate_recommendations`）时必须先展示确认卡片，用户确认后才调用 `/refine`。
- `/refine` 对应的局部重新生成 SSE `completed` 后，报告页顶部出现/更新版本切换器，可在 `v1`/`v2` 等版本间切换查看。
- 方案对比视图（手动勾选、版本对比、对话触发三种入口）渲染后至少展示学校/专业/位次差/学费/风险等级 5 个维度的逐项差异，且差异项有视觉区分。
- `/admin/debug` 路由对 `role != admin` 的用户重定向 403 页面，不渲染任何 run 数据。
- LangGraph 拓扑图中 9 个节点全部正确渲染，并行区域（retrieval + policy_rule）有浅灰背景框标注。
- 节点状态颜色随 Debug SSE 事件实时更新：`running` 状态展示脉冲动画，`completed` 展示绿色勾，`degraded` 展示橙色警告三角。
- `parallel_fan_out` 事件到达时两条并行边同时高亮，`parallel_fan_in` 到达时高亮消失。
- Reflection 回退循环箭头在 `reflection_iteration (passed=false)` 事件时闪烁一次，节点右侧显示当前迭代计数（如 `1/3`）。
- Debug 事件时间线 Auto-scroll 默认开启，新事件到达时自动滚动到底部。
- "实时跟随"开关开启后，新 run 出现时自动切换选中，拓扑图和时间线自动重置到新 run。
- Redis Stream 过期（>7天）的历史 run 点击时展示提示并跳转 LangSmith 按钮，不展示空时间线。
- 顶部系统指标条错误率 > 10% 时整条变红色背景；P95 > 60s 时数字标红。

### 12.2 体验验收

- 1280px、1440px、1728px 桌面宽度下，工作台无横向滚动、无元素溢出。
- 小于 1024px 宽度时展示桌面使用提示，不承诺移动端功能。
- 高风险项有明确视觉提示（红色标签、色块）。
- 每个关键推荐都有证据入口可点击。
- 空状态、加载状态、错误状态均有对应 UI，不展示空白页面。
- Agent 生成超过 30 秒时，对话内生成过程卡片有明确的等待引导文案。

### 12.3 合规验收

- 全流程不出现禁止文案清单中的任何词语。
- 不出现价格、套餐、订单、支付、付费解锁相关 UI 元素。
- 所有推荐结论有数据来源标注，不出现无来源的确定性断言。
