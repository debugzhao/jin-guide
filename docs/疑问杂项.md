## Q: 当前后端系统用的是什么数据库？是 PostgreSQL 吗？什么场景下会用到 PostgreSQL？为什么不用 MySQL？它是向量数据库吗？真正的企业级开发用的是什么向量数据库？

**结论**：后端主库是 PostgreSQL（单库承担关系数据 + 向量检索 + 关键词检索三重角色），不是 MySQL；PostgreSQL 本身不是专业向量数据库，而是通过 `pgvector` 扩展获得向量能力；企业级大规模场景通常用 Milvus/Pinecone/Weaviate/Qdrant 等专用向量数据库。

### 一、数据存储分工

| 存储 | 承担角色 | 依据 |
|---|---|---|
| **PostgreSQL** | 主库：用户/会话/志愿档案/报告/对话历史等业务表 + pgvector 向量检索 + pg_bm25 关键词检索 | `docker-compose.yml` 中的 `postgres` 服务，端口 5432 |
| **Redis** | 会话热层缓存、验证码存储、ARQ 任务队列、SSE 事件流 | 端口 6379，`auth:code:{email}`、`sse:{run_id}` 等 key 空间 |
| **LangGraph checkpoint 表**（`checkpoints`/`checkpoint_blobs`/`checkpoint_writes`）| Agent 执行状态持久化，由 LangGraph 框架自动管理，物理上也存在 Postgres 里 | CLAUDE.md「数据模型要点」 |

### 二、PostgreSQL 的三个用途

1. **常规关系型业务数据**：`users`、`sessions`（ORM 类名 `AuthSession`）、`admission_scores`、`province_thresholds`、`agent_runs`、`report_conversations`、`intake_conversations` 等——标准增删改查、事务、外键约束。
2. **向量检索**：`chunks` 表 + `pgvector` 扩展，存放 `text-embedding-3-small`（1536 维）的 embedding，做语义召回。
3. **关键词检索（BM25）**：PostgreSQL 原生 `tsvector` 不是真正的 BM25 算法，项目设计里需要 `pg_bm25`（ParadeDB）扩展在 `chunks` 表上建 BM25 索引，实现"BM25 + 向量"混合检索（RAG 常见架构）。

### 三、为什么不用 MySQL

| 维度 | PostgreSQL | MySQL |
|---|---|---|
| 扩展生态 | `pgvector`、`pg_bm25`（ParadeDB）、JSONB、全文检索等插件成熟，一个库能同时做向量检索 + 关键词检索 + 业务查询的联合 SQL（比如"召回向量后再 join 业务表做权限过滤"） | 8.0.32+ 才有原生 VECTOR 类型，生态和 RAG 场景成熟度明显落后于 pgvector |
| 混合检索 | 向量结果和关系数据在同一个事务/同一次 SQL 里就能 JOIN，架构简单 | 需要额外的向量存储或搜索引擎配合，跨库一致性成本更高 |
| 本项目取舍 | MVP 阶段要快速支撑 RAG，选一个能"关系数据 + 向量 + 关键词"三合一的库，运维成本最低 | 不是不能用，只是要额外再接一个向量库，增加基础设施复杂度 |

这不是"MySQL 不行"，而是 RAG/AI 场景下 Postgres 插件生态目前更适合"一库多用"，减少 MVP 阶段的基础设施数量。

### 四、PostgreSQL 是向量数据库吗

**不是**。它本质是通用关系型数据库，`pgvector` 只是给它加装了一个"向量列类型 + 近似最近邻索引（IVFFlat/HNSW）"的扩展插件，属于"关系库 + 向量能力"，而非从底层为向量检索设计的专用数据库。

### 五、企业级场景真正的专用向量数据库

| 产品 | 定位 |
|---|---|
| **Milvus / Zilliz Cloud** | 开源/云原生向量库，专为十亿级向量、高并发检索设计，国内外用得较多 |
| **Pinecone** | 全托管 SaaS 向量数据库，免运维，企业常见首选 |
| **Weaviate** | 开源，自带混合检索（向量+BM25）、GraphQL 查询、多模态支持 |
| **Qdrant** | 开源，Rust 实现，性能好，支持复杂 payload 过滤 |
| **Elasticsearch / OpenSearch** | 搜索引擎起家，加了向量检索能力，适合已有 ES 基础设施的团队 |
| **云厂商托管方案** | Azure AI Search、AWS OpenSearch Vector Engine、Google Vertex AI Vector Search |

**什么时候该从 pgvector 换成专用向量库**：向量规模到千万/亿级、需要极致的 ANN 召回性能、需要多租户隔离和水平扩展、或者向量检索已经是核心瓶颈时。另外要注意——切换 embedding 模型（如换成 BGE）时必须全库重建索引，这一点无论 pgvector 还是专用向量库都一样需要考虑。

对于问津 Agent 目前 MVP 阶段的数据规模，pgvector 够用且运维简单是合理选择。
